WHAT THIS MODEL DOES
─────────────────────
Compares a candidate's answer against a perfect ideal
answer and gives a similarity score from 0.0 to 1.0.

It does NOT just check keywords.
It understands the MEANING of sentences.

Example:
  Ideal answer   : "REST APIs use stateless HTTP
                    communication with standard verbs"
  Candidate says : "HTTP-based APIs don't store session
                    data between requests"
  Score          : 0.87  (means same thing, different words)

  Candidate says : "APIs are used to connect apps"
  Score          : 0.31  (too vague, misses the point)

This is what makes NeuralHire smarter than keyword
matching. Two answers can use completely different
words but mean the same thing — this model catches that.


THE BIG DIFFERENCE FROM CONFIDENCE SCORER
───────────────────────────────────────────
Confidence Scorer  → HOW the candidate speaks
                     (are they hesitant or sure?)

Semantic Ranker    → WHAT the candidate actually said
                     (is the content correct and complete?)

Both scores combine in the final evaluation.

Specifically: all-MiniLM-L6-v2
It was pre-trained by Microsoft on 1 billion
sentence pairs. You just use it as-is.


HOW IT WORKS (simple version)
──────────────────────────────
Step 1 : You give it two sentences
         → the ideal answer
         → the candidate's answer

Step 2 : It converts both sentences into vectors
         (lists of 384 numbers that represent meaning)

Step 3 : It measures how similar those two vectors are
         This is called cosine similarity

Step 4 : Returns a score from 0.0 to 1.0
         0.0 = completely different meaning
         1.0 = identical meaning

That's it. No training. No dataset. Just math.


WHAT ARE IDEAL ANSWERS
───────────────────────
For every interview question, you need a pre-written
ideal answer to compare against.

You generate these using Claude.

Example:
  Question     : "Explain what a REST API is"
  Ideal answer : "REST is an architectural style for
                  distributed systems using stateless
                  HTTP communication. Resources are
                  identified by URIs and manipulated
                  using standard HTTP verbs: GET for
                  retrieval, POST for creation, PUT
                  for full updates, PATCH for partial
                  updates, DELETE for removal. Each
                  request contains all information
                  needed — no server-side session state."

You need one ideal answer per question.
The system generates 8 questions per interview.
So you need 8 ideal answers per job role.

Generate ideal answers for these roles:
  → Software Engineer
  → Backend Developer
  → Frontend Developer
  → Full Stack Developer
  → System Design

That is about 40 ideal answers total.
Store them in a JSON file.
Takes about 30 minutes with Claude.

HOW TO GENERATE USING CLAUDE
  Go to claude.ai
  Send this prompt (copy paste exactly):

  ─────────────────────────────────────────────────
  Generate 8 technical interview questions for a
  Software Engineer role along with ideal answers.

  For each:
  - question: the interview question
  - ideal_answer: a detailed, technically precise
    answer that a senior engineer would give
    (3-5 sentences, uses correct terminology)

  Return as a JSON array.
  ─────────────────────────────────────────────────

  Run this prompt 5 times changing the role each time:
    → Software Engineer
    → Backend Developer
    → Frontend Developer
    → Full Stack Developer
    → System Design

  Each run gives 8 Q&A pairs.
  5 runs × 8 = 40 ideal answers total.

SAVE INTO
  ml_models\semantic_ranker\data\ideal_answers.json

    },
    {
      "role"         : "Software Engineer",
      "question"     : "What is a database index?",
      "ideal_answer" : "A database index is a data structure..."
    },

Example output:
  {
    "semantic_score" : 0.87,
    "label"          : "strong match",
    "question"       : "Explain what a REST API is",
    "ideal_used"     : "REST is an architectural style..."

Score labels:
  0.0 – 0.4  → "poor match"     (missed the point)
  0.4 – 0.6  → "partial match"  (some correct ideas)
  0.6 – 0.8  → "good match"     (mostly correct)
  0.8 – 1.0  → "strong match"   (excellent answer)

This score feeds directly into:
  → The live skill radar chart
  → The score fusion layer (30% weight)
  → The evaluation report

For context — here is how your model fits into the
bigger picture with all other models:

  Semantic Ranker score    × 30%  (what they said)
  Confidence Scorer score  × 25%  (how they said it)
  Hesitation Detector      × 20%  (how nervous they were)
  Claude evaluation        × 25%  (overall AI judgment)
  ─────────────────────────────────────────────────
  Final score              = weighted average → 0 to 10

Your model carries the highest single weight (30%)
because content accuracy matters most in an interview.
